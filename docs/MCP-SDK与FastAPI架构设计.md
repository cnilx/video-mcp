# MCP SDK 与 FastAPI 架构设计说明

## 概述

本项目采用 **MCP SDK + FastAPI** 的架构方案，实现一个可远程部署的视频分析 MCP 服务。这种架构结合了 MCP 协议的标准化能力和 FastAPI 的高性能 HTTP 服务能力。

## 核心组件

### 1. MCP SDK (mcp==1.1.2)

**作用**：实现 MCP (Model Context Protocol) 协议，提供标准化的工具接口。

**职责**：
- **协议实现**：提供 MCP 协议的 Python 实现
- **工具注册**：定义和注册 5 个视频分析工具
- **消息处理**：解析 Claude 的请求，调用相应工具，返回结构化响应
- **参数验证**：验证工具调用的参数格式和类型
- **标准化接口**：让 Claude 能够像调用内置工具一样使用这些功能

**提供的工具**：
1. `download_video` - 下载在线视频
2. `extract_audio` - 提取音频
3. `transcribe_audio` - 语音转文本
4. `extract_frames` - 提取视频帧
5. `analyze_frames` - 图像识别

### 2. FastAPI (fastapi==0.115.0)

**作用**：提供高性能的 HTTP 服务，作为 MCP 协议的传输层。

**职责**：
- **HTTP 服务器**：提供异步 HTTP/SSE 服务
- **路由管理**：定义 API 端点（/health, /mcp）
- **请求接收**：接收来自 Claude 的 HTTP 请求
- **认证中间件**：实现 Bearer Token 认证
- **长超时支持**：支持视频处理等耗时操作（默认 3600 秒）
- **错误处理**：统一的错误响应格式

**API 端点**：
- `GET /health` - 健康检查
- `POST /mcp` - MCP 协议的 HTTP 端点
- `GET /mcp/sse` - MCP 协议的 SSE 端点（可选）

### 请求处理流程

```
1. Claude 发起工具调用
   POST http://localhost:8000/mcp
   Headers: Authorization: Bearer <api_key>
   Body: MCP 协议格式的请求

2. FastAPI 接收请求
   ├─ 认证中间件验证 API Key
   ├─ 路由到 /mcp 端点
   └─ 转发给 MCP SDK

3. MCP SDK 处理请求
   ├─ 解析 MCP 协议消息
   ├─ 识别要调用的工具（如 download_video）
   ├─ 验证参数
   └─ 调用对应的工具函数

4. 工具层执行
   ├─ 调用核心处理模块
   ├─ 处理业务逻辑
   └─ 返回结构化结果

5. MCP SDK 格式化响应
   └─ 将结果封装为 MCP 协议格式

6. FastAPI 返回响应
   └─ HTTP 200 + JSON body

7. Claude 接收结果
   └─ 解析并使用工具返回的数据
```

## 两者的协作关系

### 职责分工

| 组件 | 角色 | 类比 |
|------|------|------|
| **FastAPI** | 传输层 | 门卫 + 邮递员 |
| **MCP SDK** | 协议层 | 翻译官 + 调度员 |

**FastAPI 的职责**：
- 监听端口，接收 HTTP 请求
- 验证身份（API Key 认证）
- 管理连接和超时
- 返回 HTTP 响应

**MCP SDK 的职责**：
- 理解 MCP 协议
- 管理工具注册表
- 路由到正确的工具
- 验证参数格式
- 格式化返回结果

### 数据流转

```python
# 1. FastAPI 接收 HTTP 请求
@app.post("/mcp")
async def mcp_endpoint(request: Request):
    # 2. 认证
    api_key = request.headers.get("Authorization")
    if not verify_api_key(api_key):
        raise HTTPException(401)

    # 3. 读取请求体
    body = await request.json()

    # 4. 转发给 MCP SDK
    response = await mcp_server.handle_request(body)

    # 5. 返回响应
    return response
```

```python
# MCP SDK 处理请求
@mcp_server.tool()
async def download_video(url: str):
```

## 为什么选择这个方案

### 与 stdio 方案对比

| 特性 | stdio 方案 | HTTP 方案 (FastAPI) |
|------|-----------|-------------------|
| **部署位置** | 仅本地 | 本地/局域网/云服务器 |
| **Docker 支持** | ❌ 不适合 | ✅ 完美支持 |
| **多客户端** | ❌ 每次启动新进程 | ✅ 服务常驻，共享使用 |
| **性能** | 较差（频繁启动） | 优秀（长连接） |
| **认证** | ❌ 无 | ✅ API Key 认证 |
| **监控** | ❌ 难以监控 | ✅ 可添加日志、指标 |
| **配置复杂度** | 简单 | 中等 |
| **代码复杂度** | 简单 | 中等 |

### 选择 HTTP 方案的原因

本项目选择 **MCP SDK + FastAPI** 方案的原因：

1. **远程部署需求**
   - 支持局域网团队共享
   - 支持云服务器部署
   - Claude Desktop 可以通过 URL 连接

2. **容器化部署**
   - Docker 封装所有依赖（Python、ffmpeg、yt-dlp）
   - 一键部署，环境一致
   - 易于维护和更新

3. **性能要求**
   - 视频处理是重量级操作
   - 服务常驻，避免重复初始化
   - 支持并发处理多个请求

4. **安全性**
   - API Key 认证保护服务
   - 可以配置 HTTPS
   - 可以添加访问控制

5. **可扩展性**
   - 可以添加监控、日志
   - 可以添加缓存机制
   - 可以添加管理界面

## 总结

**MCP SDK + FastAPI** 架构方案的核心优势：

1. **标准化**：MCP SDK 提供标准协议，Claude 可以无缝集成
2. **高性能**：FastAPI 提供异步 HTTP 服务，性能优秀
3. **灵活部署**：支持本地、局域网、云服务器多种部署方式
4. **易于维护**：Docker 容器化，环境一致，更新方便
5. **安全可控**：API Key 认证，可配置 HTTPS
6. **可扩展**：易于添加监控、日志、缓存等功能

这种架构既保持了 MCP 协议的标准化优势，又获得了 HTTP 服务的灵活性和可扩展性，是构建生产级 MCP 服务的理想选择。
