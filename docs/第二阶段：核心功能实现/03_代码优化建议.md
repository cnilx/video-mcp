# 下载器代码优化建议

## 当前状态评估

### 代码质量

| 维度 | 评分 | 说明 |
|------|------|------|
| 功能完整性 | ⭐⭐⭐⭐⭐ | 支持多平台，功能齐全 |

4. ✅ 过早优化是万恶之源

如果要优化，建议只做这些低风险改动：

### 1. 提取抖音解析器

```python
# src/core/douyin_resolver.py
class DouyinResolver:
    """抖音视频地址解析器"""

    @staticmethod
    async def resolve_video_url(url: str) -> Optional[tuple[str, str]]:
        """解析抖音短链，获取视频播放地址和标题"""
        # 当前 _resolve_douyin_url 的逻辑
        pass
```

### 2. 添加单元测试

特别是抖音解析逻辑的测试：

```python
# tests/test_douyin_resolver.py
async def test_resolve_douyin_url():
    resolver = DouyinResolver()
    result = await resolver.resolve_video_url("https://v.douyin.com/xxx/")
    assert result is not None
    assert len(result) == 2  # (url, title)
```

### 3. 完善错误处理

定义更明确的异常类型：

```python
class DownloadError(Exception):
    """下载错误基类"""
    pass

class PlatformNotSupportedError(DownloadError):
    """平台不支持错误"""
    pass

class VideoNotFoundError(DownloadError):
    """视频不存在错误"""
    pass
```

### 4. 提取配置常量

将 User-Agent 等常量提取到配置文件：

```python
# src/core/config.py
USER_AGENT_DESKTOP = 'Mozilla/5.0 (Windows NT 10.0; Win64; x64) ...'
USER_AGENT_MOBILE = 'Mozilla/5.0 (iPhone; CPU iPhone OS 17_0 ...'
```

## 未来重构时机

当出现以下情况时再考虑重构：

1. **需要支持第 5+ 个平台**
2. **平台特定逻辑变得复杂**
3. **出现明显的维护问题**
4. **需要支持插件化平台扩展**

## 推荐的重构方案（未来）

### 策略模式

```
src/core/
├── downloader.py          # 主下载器
├── platforms/
│   ├── __init__.py
│   ├── base.py           # 平台基类
│   ├── douyin.py         # 抖音平台处理
│   ├── youtube.py        # YouTube 平台处理
│   └── bilibili.py       # Bilibili 平台处理
```

优点：
- 每个平台独立维护
- 易于测试
- 易于扩展新平台
- 符合开闭原则

缺点：
- 代码文件增多
- 需要重构现有代码

## 总结

**当前阶段**: 保持现状，做小优化

**原因**:
- 当前代码质量良好，无需立即重构
- 只支持 3 个平台，复杂度可控
- 避免过度设计

**后续规划**: 当支持平台超过 5 个时，考虑策略模式重构
